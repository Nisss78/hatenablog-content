<h3>第32章：throw文の爆発</h3>

<p>try-catchの試練を乗り越えたカイトたちは、地下実験場のさらに奥へと進んでいた。石造りの廊下は次第に広くなり、壁に刻まれた古代文字が青白い光を放ち始めていた。</p>

<p>「ねえ、カイト」エリナが歩きながら声をかけてきた。「さっきのtry-catch、すごかったわね。あんなに複雑なエラーを次々と処理するなんて」</p>

<p>カイトは肩をすくめた。「プログラミングの世界じゃ当たり前の技術だよ。エラーは必ず起こる。大事なのは、どう対処するかだ」</p>

<p>「実に建設的な思考だ」リアムが頷いた。「我々の世界でも、失敗を想定して準備する者は少ない。大抵は成功を信じて突き進み、失敗した時に初めて慌てる」</p>

<p>セシリアが後ろで小さく笑った。「私の父もそうでしたわ。計画性より勢いを重んじる types で」</p>

<p>廊下の突き当たりに、巨大な石の扉が現れた。表面には複雑な魔法陣が刻まれ、中央には一つの単語が浮かび上がっていた。</p>

<p><strong>throw</strong></p>

<p>カイトは眉をひそめた。「throw？ 例外を投げる構文か」</p>

<p>「投げる？」エリナが首をかしげた。「何を投げるの？」</p>

<p>「エラーだよ」カイトは説明した。「try-catchはエラーを捕まえる仕組みだけど、throwは逆に自分からエラーを発生させるんだ。例えば、予期せぬ状況が起きた時に、『これは想定外だ』って叫ぶようなものさ」</p>

<p>「自分からエラーを起こすの？」リアムは困惑した表情を浮かべた。「何のためにそんなことを？」</p>

<p>「状況に応じて処理を中断したい時がある。例えば……」カイトは考えた。「医者が手術中に『この患者は手術不可能だ』と判断したら、途中で止めるだろ？ それと同じだ。無理に続けるより、明確に『できない』と宣言して、適切な対処に委ねる方が安全なんだ」</p>

<p>「なるほど」セシリアが理解したように頷いた。「魔法で言えば、詠唱を途中で中断して、より安全な手段に切り替えるようなものですわね」</p>

<p>カイトは石の扉に近づいた。「ここにも試練があるんだろうな」</p>

<p>扉に触れた瞬間、魔法陣が激しく輝いた。</p>

<pre><code>class TrialThrow extends Trial {
    constructor() {
        super("例外を意図的に投げよ");
    }
    
    execute() {
        // 条件を満たさない時は例外を投げる
        if (!this.validate()) {
            throw new Error("条件不足");
        }
        return this.proceed();
    }
}</code></pre>

<p>光が収まると、扉がゆっくりと開いた。その向こうには、今までとはまったく異なる空間が広がっていた。</p>

<p>広大な円形の部屋。床には幾何学的な模様が描かれ、中央には巨大な水晶が浮かんでいた。しかし、水晶はひび割れており、そこから黒い霧が漏れ出していた。</p>

<p>「なんだこれは」カイトは警戒して身構えた。</p>

<p>黒い霧が凝縮され、人の形をとり始めた。やがて、古代の魔法使いの姿をした幻影が現れた。長い髭、尖った帽子、手には杖。</p>

<p>「訪問者よ」幻影が語りかけた。声は重く、響き渡った。「我はヴェリディアン。この実験場の管理者だった者だ」</p>

<p>「管理者？」エリナが前に出た。「この実験場で何をしていたの？」</p>

<p>「魔法の本質を解明しようとしていた」ヴェリディアンは答えた。「だが、一つの過ちを犯した。try-catchだけでは防げないエラーの存在に気づかなかったのだ」</p>

<p>カイトは首をかしげた。「try-catchで防げないエラー？ そんなものがあるのか？」</p>

<p>「ある」ヴェリディアンは頷いた。「自分で定義したエラーだ。予期せぬ状況を自ら作り出し、システム全体に警告を発する。throw文の本質は、単なるエラー処理ではない。状況を能動的に変化させる力なのだ」</p>

<p>幻影は四人を見渡した。「君たちには、この部屋の守護者を鎮めてもらう。守護者は、特定の条件でのみthrow文を発動するよう設計されている。条件を見極め、適切な例外を投げよ。さもなくば、この部屋から出られん」</p>

<p>言い終わると同時に、ヴェリディアンの幻影は霧散した。</p>

<p>床の模様が光り始めた。中央の水晶から、新たな実体が現れた。鎧を着た騎士の形をしたエネルギー体だ。手には剣を持っているが、その剣は半透明で、絶えず形を変えていた。</p>

<p>「throwの守護者だ」カイトは分析した。「特定の条件で例外を投げるようプログラムされている」</p>

<p>騎士が動き出した。剣を振り上げ、カイトたちに向かって突進してくる。</p>

<p>「分散！」リアムが叫んだ。</p>

<p>四人は四方に散った。騎士の剣が床を叩き、衝撃波が走った。</p>

<p>「カイト、何か見えた？」エリナが魔法で防御壁を展開しながら叫んだ。</p>

<p>カイトは騎士を観察していた。「動きにパターンがある。剣を振る瞬間、胸のプレートが光る。あそこに条件が書かれているはずだ」</p>

<p>騎士が再び攻撃を仕掛けてきた。カイトは身を翻して避け、正面から騎士の胸元を覗き込んだ。</p>

<p>そこには、古代文字で条件が刻まれていた。</p>

<pre><code>if (opponent.power > 100 && opponent.hasWeapon) {
    throw new CriticalError("圧倒的な力を検知");
}</code></pre>

<p>「読めた！」カイトは叫んだ。「相手の力が100を超えて、かつ武器を持っている時に例外を投げる仕組みだ！」</p>

<p>「つまりどういうこと？」セシリアが魔法の矢を放ちながら尋ねた。</p>

<p>「この条件を満たせば、守護者は自分からエラーを起こして停止するはずだ。でも……」カイトは顔をしかめた。「俺たちの力が100を超える必要がある。どうやって測るんだ？」</p>

<p>「私が！」リアムが前に出た。「私の聖剣があれば、条件を満たせるかもしれない」</p>

<p>リアムは腰の剣を抜いた。刀身が白い光を放つ。彼は騎士に向かって突進した。</p>

<p>剣と剣が交錯した。火花が散る。</p>

<p>「くっ、重い」リアムは歯を食いしばった。</p>

<p>騎士は動じなかった。むしろ、リアムの力を吸収するように、剣の光が強くなった。</p>

<p>「ダメだ」カイトは気づいた。「この条件は罠だ。相手の力が100を超えたら例外を投げるんじゃない。そうなったら攻撃を強化するんだ」</p>

<p>「何ですって？」エリナが叫んだ。</p>

<p>「コードの意味を履き違えていた」カイトは必死に考えた。「throwは自分からエラーを投げる。この守護者は、強い敵と戦う時に、自分の限界を超えるためにthrowを使うんだ」</p>

<p>「つまり？」セシリアが避難しながら尋ねた。</p>

<p>「条件を満たすと、守護者は限界突破するってことだ。逆に危険になる」</p>

<p>騎士が咆哮を上げた。リアムを弾き飛ばし、カイトの方へ向き直った。先ほどより動きが鋭くなっている。</p>

<p>「違う方法を考える必要がある」カイトは思考を巡らせた。「条件を満たさずに、守護者を停止させるには……」</p>

<p>その時、カイトの脳裏にアイデアが闪いた。</p>

<p>「throwの本質は、自分で例外を定義することだ。相手の条件を利用するんじゃなくて、俺たち自身の例外を作るんだ」</p>

<p>「どうやって？」エリナが尋ねた。</p>

<p>カイトは騎士に向かって歩き出した。「コードを書くんだ。新しい例外を」</p>

<p>彼は空中に指で文字を描いた。光の軌跡が残る。</p>

<pre><code>class GuardianOverload extends Error {
    constructor(message) {
        super(message);
        this.name = "GuardianOverload";
    }
}

function triggerGuardian(guardian) {
    throw new GuardianOverload("守護者の論理矛盾を検知");
}</code></pre>

<p>「カスタム例外だ」カイトは説明した。「標準のエラーじゃなくて、自分で定義した特別なエラーを投げる。これなら、守護者の予想外の動作を引き起こせるかもしれない」</p>

<p>騎士が剣を振り上げた。カイトは避けなかった。代わりに、描いたコードを騎士の方へ押し出した。</p>

<p>「trigger Guardian Overload」</p>

<p>光のコードが騎士に命中した瞬間、異変が起きた。騎士の動きが止まった。胸のプレートが明滅し始める。内部で何かが衝突しているようだった。</p>

<p>「エラーの競合だ」カイトは解説した。「守護者は『力が100以上で武器あり』の条件で例外を投げるよう設計されている。でも、俺が投げたカスタム例外が、その条件と衝突している。守護者のシステムが、どちらの例外を優先すべきか判断できなくなっている」</p>

<p>騎士が苦しむような動きを見せた。剣が震え、体が揺らぐ。</p>

<p>「今だ！」リアムが叫んだ。「カイト、もう一度、別の例外を投げてくれ。システムを完全にオーバーロードさせるんだ」</p>

<p>カイトは頷いた。再び空中に文字を描いた。</p>

<pre><code>try {
    guardian.execute();
} catch (GuardianOverload) {
    throw new SystemCrash("論理ループを検知");
} finally {
    guardian.shutdown();
}</code></pre>

<p>「try-catch-throwの連鎖だ」カイトは言った。「最初の例外をcatchして、そこから別の例外を投げる。finallyで強制終了させる」</p>

<p>コードが騎士に吸い込まれた。</p>

<p>騎士が叫び声を上げた——初めて音を発した。体中の光が乱れ、形が崩れ始めた。</p>

<p>「例外の連鎖が、内部ループを引き起こした」カイトは息を切らしながら説明した。「守護者は例外を処理しようとして、その処理の中でまた例外が発生する。無限ループだ」</p>

<p>騎士が崩壊した。光の粒子となって四散した。</p>

<p>中央の水晶のひび割れが修復され、黒い霧が消えた。</p>

<p>静寂が戻った。</p>

<p>「やったわね」エリナがカイトに駆け寄った。「自分で例外を作って、相手のシステムをクラッシュさせるなんて」</p>

<p>カイトは肩をすくめた。「throwの本質だよ。受け身じゃなくて、能動的に状況を変える力。プログラミングでは、予期せぬ事態に自分でエラーを投げて、処理の流れを変えることがある。魔法でも同じことができるんだな」</p>

<p>リアムが剣を納めた。「驚いたよ。カイト、君の世界の技術は、単なる道具じゃない。戦術そのものだ」</p>

<p>「便利でしょ」カイトは苦笑した。</p>

<p>セシリアが水晶の元へ歩いた。「この水晶、修復されたようですわ。何か情報が残されているかもしれません」</p>

<p>水晶に触れると、再びヴェリディアンの声が響いた。</p>

<p>「見事だ。throw文の試練を乗り越えた。君たちは、状況を能動的に変化させる力を理解した。だが、これは始まりに過ぎない。実験場の最深部には、さらなる試練が待っている。次は、『非同期の試練』だ」</p>

<p>「非同期？」カイトは首をかしげた。「async/awaitのことか？」</p>

<p>「答えは現地で見つけるがよい」</p>

<p>声が消えると、部屋の奥に新たな通路が現れた。</p>

<p>「非同期処理か」カイトは呟いた。「時間差で処理を実行する技術。この世界でどう表現されるんだろう」</p>

<p>「行けばわかるわ」エリナが微笑んだ。「これまでの試練も、全部新しい発見だったもの」</p>

<p>四人は通路へと歩き出した。</p>

<p>カイトは考えながら歩いた。throw文の応用——カスタム例外の作成、例外の連鎖、finallyとの組み合わせ。この世界の魔法は、プログラミングの概念を驚くほど忠実に再現していた。</p>

<p>（次は非同期か。時間を操る魔法なのか、それとも並行処理なのか。どちらにせよ、面白くなりそうだ）</p>

<p>地下の闇が、四人を深く深く、飲み込んでいった。</p>

<hr>

<p><strong>【プログラミング解説】</strong></p>

<ul>
<li><strong>throw文</strong>：自分で意図的に例外を投げる構文。特定の条件で処理を中断したい時に使用する。</li>
<li><strong>カスタム例外</strong>：Errorクラスを継承して、独自の例外を定義できる。状況に応じたエラー処理が可能になる。</li>
<li><strong>例外の連鎖</strong>：catchブロックの中で別の例外を投げることで、処理の流れを連続的に変化させられる。</li>
<li><strong>finally</strong>：try-catchの後に続くブロックで、例外の有無に関わらず必ず実行される。クリーンアップ処理に使う。</li>
</ul>
